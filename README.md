# Тестовое задание "Кредитный Конвейер"

## 1. Стратегия решения задачи
В качестве среды тестирования выбран подход **Data-Driven Testing (DDT)**. Тестовый набор данных вынесен в отдельный файл `test_data.csv`, содержащий все комбинации входных значений и ожидаемый результат. Это позволяет легко масштабировать тесты без необходимости изменения Python-кода.
В рамках тест-дизайна использовались следующие техники:
- **Decision Table (Таблица принятия решений)**: для покрытия всех бизнес-правил, их пересечений и стоп-факторов.
- **Boundary Value Analysis (Анализ граничных значений)**: тестирование границ возраста (17, 18, 20, 21, 100, 101) и дохода (-1, 0, 1000000, 1000001).
- **Error Guessing (Предугадывание ошибок)**: тестирование устойчивости процедуры к `NULL` и невалидным перечислениям (invalid enum), чтобы проверить прочность "черного ящика".

Механизм автотестирования реализован на стеке **Python 3.10+ + Pytest + Allure**. Для валидации логики был написан DDL-скрипт `init.sql`, поднимающий локальную мок-версию хранимой процедуры `SP_GET_CREDIT_DECISION` в PostgreSQL через Docker.

## 2. Оценка корректности постановки задачи (Баг-репорт)
В ходе ревью требований (ТЗ) были выявлены следующие неточности:

1. **Конфликт бизнес-правил (Priority Conflict)**:
   - *Исключение из Правила 1* гласит: Возраст 18-20 И Доход > 50 000 И Занятость = 'EMPLOYED' -> **ОДОБРЕНО**.
   - *Правило 2* гласит: При кредитной истории 'BAD' -> **ОТКАЗ всегда**.
   - **Проблема**: В ТЗ не указан приоритет этих правил. Что будет, если придет клиент 19 лет, трудоустроенный, с доходом 60к, но с кредитной историей BAD? В текущей реализации (mock SQL и автотестах) принято допущение из банковской практики: плохая кредитная история является жестким стоп-фактором (Knock-out rule) и перекрывает исключения по возрасту. Возвращается Reject. Но это необходимо обсудить с аналитиками.
2. **Отсутствие логики для невалидных данных (Robustness)**:
   - В ТЗ сказано про "Одобрено/Отказ", но не описано, как система должна реагировать на абсолютно некорректные данные (например, возраст 17, возраст 1500, отрицательный доход, пустые значения (`NULL`)).
   - По стандарту такие данные должны возвращать 400 ошибку еще на уровне API. Если они доходят до БД, мы заложили в нашей процедуре генерацию исключения `VALIDATION_ERROR`, что является более безопасным подходом, чем возврат стандартного `Reject` (не тратим время скоринговой модели). 

## 3. Инструкция по запуску автотестов
1. **Подготовка окружения**:
   Убедитесь, что у вас работает Docker и установлен Python (рекомендуется `uv`). 
   Запустите тестовую базу данных PostgreSQL с развернутой хранимой процедурой:
   ```bash
   docker-compose up -d
   ```
   *(DB поднимется на порту 5433 для избежания конфликтов)*

2. **Запуск тестов**:
   Через установщик зависимостей нового поколения `uv`:
   ```bash
   uv sync
   uv run pytest test_credit_decision.py --alluredir=allure-results -v
   ```

3. **Генерация отчета Allure**:
   Для просмотра оформленного отчета необходима утилита Allure CLI. 
   Установите её (Mac: `brew install allure`, для других ОС используйте [инструкцию](https://allurereport.org/docs/gettingstarted/installation/)).
   После этого выполните:
   ```bash
   allure serve allure-results
   ```
