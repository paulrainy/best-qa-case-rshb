# Тестовое задание «Кредитный Конвейер»

## 1. Стратегия решения задачи

Задача требует протестировать «чёрный ящик» — хранимую процедуру `SP_GET_CREDIT_DECISION`, реализующую кредитный скоринг. Для этого был выбран следующий путь.

### Шаг 1: Ревью требований

Перед написанием тестов было проведено **ревью ТЗ** — проверка бизнес-правил на полноту и непротиворечивость. Это позволило сразу выявить дефекты в требованиях (см. раздел 2) и принять явные допущения до начала реализации.

### Шаг 2: Тест-дизайн

Для систематического покрытия всех бизнес-правил применялись три техники тест-дизайна в связке:

**Decision Table** — все правила и их пересечения сведены в матрицу:

| # | Age < 21 | Income > 50k | Emp = EMPLOYED | History = BAD | Emp = UNEMPLOYED | Результат |
|---|---|---|---|---|---|---|
| 1 | – | – | – | Да | – | REJECT |
| 2 | – | – | – | – | Да | REJECT |
| 3 | Да | Нет | – | Нет | Нет | REJECT |
| 4 | Да | – | Нет | Нет | Нет | REJECT |
| 5 | Да | Да | Да | Нет | – | APPROVE |
| 6 | Нет | – | – | Нет | Нет | APPROVE |

Правила «BAD history» и «UNEMPLOYED» — это **Knock-out rules** (стоп-факторы), применяющиеся в первую очередь.

**Boundary Value Analysis (BVA)** — ошибки реализации чаще всего скрываются на границах (`<` vs `<=`), поэтому для каждого числового параметра проверены тройки граница−1 / граница / граница+1:

| Граница | Значения тестирования | Ожидаемое поведение |
|---|---|---|
| Age нижняя (18) | 17 → VALIDATION_ERROR, 18 → APPROVE (при исключении) | Правило 1 / Валидация |
| Age исключения (21) | 20 → APPROVE (если доход > 50к), 21 → APPROVE (стандарт) | Правило 1 |
| Age верхняя (100) | 100 → APPROVE, 101 → VALIDATION_ERROR | Валидация |
| Income исключения (50 000) | 50 000 → REJECT, 50 001 → APPROVE (для 18–20) | Исключение правила 1 |
| Income верхняя (1 000 000) | 1 000 000 → APPROVE, 1 000 001 → VALIDATION_ERROR | Валидация |
| Income нижняя (0) | −1 → VALIDATION_ERROR, 0 → APPROVE | Валидация |

> **Ключевой момент:** условие исключения `> 50 000` (строго больше), значит ровно 50 000 — это REJECT. Кейс `case_id=8` (income=50 001) специально покрывает эту границу.

**Equivalence Partitioning** — для каждого параметра выделены классы, дающие одинаковый результат:

| Параметр | Валидные классы | Невалидные классы |
|---|---|---|
| Age | 21–100, 18–20 (с исключением) | < 18, > 100, NULL |
| Income | 0–50 000 (без исключения), 50 001–1 000 000 | < 0, > 1 000 000, NULL |
| CreditHistory | GOOD, EMPTY | BAD, anything else, NULL |
| Employment | EMPLOYED, SELF_EMPLOYED | UNEMPLOYED, anything else, NULL |

**Результирующие классы:** `APPROVE`, `REJECT`, `VALIDATION_ERROR`.

**Error Guessing** — поверх формальных методов добавлены кейсы из опыта: `NULL`-значения (случаи 16–19), отрицательный доход, невалидный enum (`CreditHistory = 'AWESOME'`), конфликтный кейс age=19 + income=60k + EMPLOYED (Emp) + BAD (History).

### Шаг 3: Приоритизация тест-кейсов

| Приоритет | Тип кейса | Примеры |
|---|---|---|
| Высокий | Основные бизнес-правила | BAD history → REJECT, UNEMPLOYED → REJECT |
| Высокий | Граничные значения возраста и дохода | 17, 18, 20, 21, 50 000, 50 001 |
| Средний | Конфликты правил | case_id 6 (BAD + исключение возраста) |
| Средний | Robustness / Validation | NULL, out-of-range, invalid enum |

### Шаг 4: Реализация

Выбран подход **Data-Driven Testing (DDT)**: логика теста (один сценарий `test_sp_get_credit_decision`) отделена от данных (`test_data.csv`, 19 кейсов). Новые тест-кейсы добавляются без изменения кода — CSV передаётся аналитику для валидации.

Стек: **Python 3.10+ + Pytest + Allure**. Мок-версия `SP_GET_CREDIT_DECISION` поднята в PostgreSQL через Docker (`init.sql`) — это обеспечивает тестирование без доступа к продакшн-базе.

### Шаг 5: Анализ результатов и отчетность

В качестве инструмента отчетности выбран **Allure Report**. 
После прохождения автотестов генерируется наглядный отчет, который отражает статус выполнения каждого тест-кейса с привязкой к исходному ТЗ. Данный подход гарантирует **traceability** (прослеживаемость): от бизнес-правила (или требования) до конкретного прогона теста. Если в бизнес-логике хранимой процедуры будут найдены дефекты, это мгновенно отразится в отчете.

## 2. Оценка корректности постановки задачи (Баг-репорт)

В ходе ревью требований выявлены следующие дефекты ТЗ:

1. **Конфликт бизнес-правил (Priority Conflict)**:
   - *Исключение из Правила 1* гласит: Возраст 18–20 И Доход > 50 000 И Занятость = `EMPLOYED` → **ОДОБРЕНО**.
   - *Правило 2* гласит: При кредитной истории `BAD` → **ОТКАЗ всегда**.
   - **Проблема**: В ТЗ не указан приоритет этих правил. Что будет, если придёт клиент 19 лет, трудоустроенный, с доходом 60к, но с кредитной историей BAD? В текущей реализации принято допущение из банковской практики: плохая кредитная история является жёстким стоп-фактором (**Knock-out rule**) и перекрывает исключения по возрасту. Возвращается `Reject`. Но это необходимо обсудить с аналитиками.

2. **Отсутствие логики для невалидных данных (Robustness)**:
   - В ТЗ описано только «Одобрено/Отказ», но не указано, как система должна реагировать на некорректные данные (возраст 17, возраст 1500, отрицательный доход, `NULL`).
   - По стандарту такие данные должны возвращать ошибку валидации ещё на уровне API. Если они доходят до БД, в нашей процедуре заложена генерация исключения `VALIDATION_ERROR` — это более безопасный подход, чем возврат стандартного `Reject`.

## 3. Инструкция по запуску автотестов

1. **Подготовка окружения**:
   Убедитесь, что у вас работает Docker и установлен Python (рекомендуется `uv`).
   
   Создайте файл с переменными окружения из шаблона:
   ```bash
   cp .env.example .env
   ```

   Запустите тестовую базу данных PostgreSQL с развёрнутой хранимой процедурой:
   ```bash
   docker-compose up -d
   ```
   *(DB поднимется на порту 5433 для избежания конфликтов)*

2. **Запуск тестов**:
   Через установщик зависимостей нового поколения `uv`:
   ```bash
   uv sync
   uv run pytest test_credit_decision.py --alluredir=allure-results -v
   ```

3. **Генерация отчёта Allure**:
   Для просмотра оформленного отчёта необходима утилита Allure CLI.
   Установите её (Mac: `brew install allure`, для других ОС используйте [инструкцию](https://allurereport.org/docs/gettingstarted/installation/)).
   После этого выполните:
   ```bash
   allure serve allure-results
   ```
